# CANONICAL_ALIGNMENT.md

Canonical alignment control for the Invariant Systems website repository.

This file exists to prevent public website wording from drifting away from the authoritative research, methodological, and claim boundaries defined in the canonical StructuralOrigins repository.

---

## 1. Purpose

The website repository is a public expression layer.
It is not the source of truth for theory, methodology, evidence ceilings, service expansion, or research claims.

Its function is to express approved public-facing business language without overstating capabilities, weakening boundaries, or outrunning canonical authority.

---

## 2. Authority Hierarchy

For all claim-affecting website content, follow this authority order:

1. direct user instruction in the active task
2. `AGENTS.md` in this repository
3. this `CANONICAL_ALIGNMENT.md`
4. the canonical source materials in the `StructuralOrigins` repository
5. current website wording

If website wording conflicts with canonical source material, the canonical source material governs.

---

## 3. Canonical Source Repository

The authoritative source for research, methodological boundaries, and claim ceilings is:

- `Christopher-Kokkinos/StructuralOrigins`

The website repository must not silently redefine, widen, compress, or soften canonical positions established there.

The website may translate canonical material into restrained public-facing language, but it must not alter underlying meaning.

---

## 4. Approved Website Claim Envelope

Unless explicitly approved by the user and supported by canonical source authority, website wording must remain within the following envelope:

- Invariant Systems is a diagnostic-only practice
- website wording may describe structural diagnosis, regime classification, and boundary identification
- website wording may describe a behavioural-evidence posture
- website wording may describe bounded engagements
- website wording may describe explicit exclusions and fit constraints

Website wording must not imply:

- optimisation services
- implementation services
- performance guarantees
- predictive guarantees
- model-internal inspection as standard scope
- certification, regulatory, or compliance authority
- broader scientific or operational claims than those canonically supported

---

## 5. Disallowed Drift

The following forms of website drift are not allowed without explicit canonical support and user approval:

- claim inflation
- softening of exclusions
- removal of epistemic caveats where those caveats are claim-bearing
- conversion of diagnostic language into consulting or implementation language
- conversion of bounded methodological wording into broad commercial promises
- introducing causal, predictive, reliability, or assurance claims not canonically grounded
- implying a wider evidence base than is actually established

---

## 6. Alignment Audit Triggers

A canonical alignment audit is required after any phase that changes:

- homepage positioning copy
- service description copy
- method page wording
- boundaries page wording
- eligibility page wording
- legal wording that touches service scope
- new pages introducing capability, service, or methodology language
- any text that could change public interpretation of what the business does

A canonical alignment audit is not required for purely non-claim changes such as:

- CSS/layout fixes
- spacing adjustments
- metadata-only edits
- favicon/image updates
- footer-only naming consistency
- file cleanup with no wording change

---

## 7. Audit Questions

Each canonical alignment audit should check for:

- overstatement
- claim inflation
- boundary weakening
- methodological drift
- evidence overreach
- wording that outruns canonical authority
- wording that changes public interpretation of scope

---

## 8. Audit Output Categories

Each audit should classify website statements into four groups:

1. aligned
2. questionable
3. misaligned
4. recommended corrections

Audits should be diagnostic first.
They should not rewrite content unless explicitly instructed to do so.

---

## 9. Safe Default

If uncertain whether proposed website wording is canonically supported:

- do not widen the claim
- do not soften the boundary
- do not infer support from tone or implication
- preserve the narrower wording
- flag the uncertainty explicitly

When in doubt, the website must remain more conservative than the canonical source, not less.

---

## 10. Operational Rule

The website repository is a controlled public surface layered on top of the canonical StructuralOrigins repository.

It may summarize.
It may translate.
It may simplify carefully.

It must not redefine.
